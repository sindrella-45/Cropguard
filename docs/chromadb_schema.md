# ChromaDB Schema Documentation

## Collection Overview

CropGuard AI uses a single ChromaDB collection
called "crop_diseases" to store all disease
knowledge as vector embeddings.

## Why Single Collection

All disease knowledge lives in one collection
making cross-document similarity search simple.
Documents are differentiated by source_file
metadata allowing filtered searches if needed.

## Collection Schema

### Collection Metadata
```json
{
  "name": "crop_diseases",
  "description": "Crop disease knowledge base",
  "embedding_model": "text-embedding-3-small",
  "chunk_size": 500,
  "chunk_overlap": 50
}
```

### Document Schema (per chunk)

Each document stored in ChromaDB represents
one chunk of text from a source PDF.
```json
{
  "id": "cabi_crop_diseases.pdf_chunk_023",
  "document": "early blight is caused by alternaria...",
  "metadata": {
    "source_file": "cabi_crop_diseases.pdf",
    "chunk_index": 23,
    "page_number": 14,
    "chunk_size": 487
  },
  "embedding": [0.023, -0.145, 0.891, ...]
}
```

### Field Descriptions

| Field | Type | Description |
|-------|------|-------------|
| id | string | Unique chunk identifier |
| document | string | Normalised chunk text |
| source_file | string | Original PDF filename |
| chunk_index | int | Position in document |
| page_number | int | Source page for citation |
| chunk_size | int | Character count |
| embedding | float[] | 1536-dimension vector |

## ID Naming Convention

All chunk IDs follow this pattern:
```
{source_filename}_chunk_{index}
```

Examples:
```
cabi_crop_diseases.pdf_chunk_0
cabi_crop_diseases.pdf_chunk_1
fao_plant_health.pdf_chunk_0
uganda_crop_guide.pdf_chunk_42
```

This naming makes it easy to:
- Identify which document a chunk came from
- Delete all chunks from a specific document
- Order chunks within a document

## Querying The Collection

### Basic Similarity Search
```python
from rag.vectorstore.collections import get_collection

collection = get_collection()

results = collection.query(
    query_texts=["yellow spots on tomato leaves"],
    n_results=3,
    include=["documents", "metadatas", "distances"]
)
```

### Filtered Search By Source
```python
# Search only within CABI documents
results = collection.query(
    query_texts=["early blight symptoms"],
    n_results=3,
    where={"source_file": "cabi_crop_diseases.pdf"}
)
```

### Get Collection Stats
```python
from rag import RAGPipeline

pipeline = RAGPipeline()
stats = pipeline.get_collection_stats()
print(stats["total_chunks"])
```

## Storage Location

ChromaDB data is stored on disk at:
```
backend/chroma_db/
```

This directory is created automatically when
the first document is ingested. It is excluded
from git via .gitignore because it can be
regenerated by running the ingestion script.

## Scalability Notes

For this project scale (3-5 PDFs):
- Local ChromaDB is perfectly sufficient
- No external service needed
- Data persists between server restarts

For production scale (100+ documents):
- Consider ChromaDB Cloud or Pinecone
- Add collection sharding by crop type
- Implement document versioning
- Add metadata filtering for regions
